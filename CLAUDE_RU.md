# Crusade Deck

Мультиплеерная карточная игра для мобильных браузеров. 2–32 игрока в сессии, колода 36
или 52 карты. Пиксельный кэжуал-стиль (Balatro-style) — не литеральный «стол с людьми»,
виртуальный круглый стол, места посадкой «П».

Правил конкретной карточной игры в коде нет — реализована вся физическая механика стола
(колода, раздача, руки, голосования), поверх которой правила можно задавать конфигурацией.

## Стек

- **Сервер**: Colyseus (Node.js, `@colyseus/schema` v2), свои аккаунты вместо Firebase.
- **Клиент**: React + Vite. Стол рисует **Pixi.js v8** императивным движком (не
  `@pixi/react` — см. ниже почему); меню и статичный UI — обычный React, Framer Motion
  только там.
- **Auth**: свои аккаунты (короткий recovery-код, без пароля). Firebase — задел на
  будущее в коде (`client/src/firebase.ts`, `server/src/auth.ts`), не настроен и не
  используется, пока не подставлены ключи.
- **Тесты**: vitest в обоих пакетах (`npm test` в `server/` и `client/`).
- **Деплой**: см. `DEPLOY_RU.md` (Cloudflare Tunnel, без открытых портов).

## Команды

```bash
cd server && npm test && npx tsc --noEmit   # 222 теста
cd client && npm test && npx tsc --noEmit   # 828 тестов
cd client && npx vite build                 # прод-сборка
```

`server/vitest.config.ts` ограничивает прогон каталогом `src/` — без этого vitest
подхватывал ещё и скомпилированные `dist/*.test.js` после `npm run build`, и два теста
`CardRoom` дрались за один тестовый порт. Тесты комнаты разрезаны по темам
(`CardRoom.deck.test.ts`, `.hands.`, `.visibility.`, `.free.`, `.votes.`, `.lifecycle.` и
`TestRoom.test.ts`) и делят общую обвязку `roomHarness.ts`. У каждого файла СВОЙ порт
(`TEST_PORTS`): vitest гоняет файлы параллельно, а `boot(server, port)` из
`@colyseus/testing` порт молча игнорирует, если передать ему готовый `Server`.

cwd в Bash-туле имеет свойство «уезжать» в корень репозитория между вызовами — перед
`tsc`/`vitest`/`vite` всегда переходи в `client/` или `server/` явно.

## Архитектура стола (клиент)

`client/src/game/RoomEngine.ts` (~3400 строк, ~187 методов по 18 строк в среднем) —
императивный движок: владеет одним Pixi
`Application`, тикером и всеми визуальными объектами (`CardVisual` — простые мутируемые
структуры, не React-ноды). `RoomCanvas.tsx` — тонкий React-хост: монтирует движок один раз,
дальше прокидывает каждый проп через `useEngineEffect` (одна строка на связь), а сразу
после mount заливает всё разом через `applyAllToEngine`. `RoomScreen.tsx` собирает
состояние комнаты (`room/useRoomState.ts`), серверные события (`room/useRoomSignals.ts`),
отступы HTML-панелей (`room/useInsets.ts`) и автораздачу (`room/useAutoDeal.ts`).

Всё, что движок делает, но что НЕ является его состоянием, живёт в
`client/src/game/engine/`, с тестами рядом: `constants.ts` (размер текстуры, палитра,
zIndex слоёв), `cardTextures.ts` (фабрики лица/рубашки/тени), `faceTextureCache.ts`
(кэш и прогрев порциями), `fanGeometry.ts`, `seatChrome.ts`/`seatPaint.ts` (правила и
отрисовка чужих мест), `zoneChrome.ts`/`zonePaint.ts` (то же для дроп-зон),
`collapseArrow.ts`, `scramble.ts`, `idleGate.ts`, `shadowPass.ts` (ОДИН проход теней на
все слои — раньше их рисовали три конкурирующих механизма), `shufflePose.ts`, `shout.ts`
(пузыри кричалок «соснуть»/«сосать»).

Почему не `@pixi/react`: прошлая попытка на нём падала под React StrictMode (двойной mount
на канвасе с уже уничтоженным WebGL-контекстом → «context lost»). Текущий движок сам создаёт
свежий канвас на каждый `mount()`.

Рендер-цикл спит в простое (`wake()`/`sleep()`) — рисует только когда что-то движется.
Условие сна — `canSleep()` в `engine/idleGate.ts`: там все активные анимации перечислены
явными полями типа. Любая новая непрерывная анимация обязана попасть в `EngineActivity`,
иначе она либо не проигрывается (цикл уснёт под ней), либо не даёт движку уснуть и жжёт
CPU/GPU вхолостую.

У движка есть сетка безопасности: `RoomEngine.test.ts` монтирует его headless на фейке
Pixi (`src/test/pixiFake.ts`, `vi.mock("pixi.js")`) и проверяет структурные инварианты —
спрайт на карту, отсутствие дубликатов, переиспользование спрайтов при тасовке, сон и
пробуждение цикла, чистый `destroy()`. Настоящий Pixi в jsdom не поднимется (нет WebGL).
Учти: цикл вообще способен уснуть только на «умеренном» профиле анимаций — на «полном»
idle-дыхание карт держит его бодрым по замыслу.

Колода и своя рука — одна и та же вещь с разной раскладкой, и обе это `engine/CardPile`:
порядок плюс спрайты, привязанные к **идентичности карты**, а не к индексу массива —
поэтому тасовку/реордер можно проиграть по-настоящему (каждая карта летит из старого слота
в новый), а не телепортировать. Вся разница между стопками — два колбэка: где лежит карта
с номером i и что сделать с только что созданным спрайтом (z-порядок у колоды, «не
вспыхивать, пока летит» у руки).

Кадр читается как оглавление: `stepPhysics` (сабстепы: сумбур, выплеск, полёты, тасовка,
пружины) → `stepFanWiggle` → `stepDraggedCard` → `stepFlipAnim` → `stepOverlays` →
`syncScene` → `maybeSleep`. `mount` так же разложен на `buildLayers` / `buildOverlays` /
`buildShadows` / `buildHitAreas` / `bindStageEvents`.

Что ЗНАЧИТ движение пальца — чистая функция `pressIntent` в `engine/gestureIntent.ts`:
`wait | deal | collapse-hand | shuffle | glissando | grab`. Именно здесь легче всего
ошибиться — например, принять медленное ведение за свайп и тасануть колоду посреди
разглядывания карт.

Игровая математика вынесена из движка в маленькие чистые модули `client/src/game/*.ts`,
каждый с параллельным `*.test.ts` — движок их только вызывает и рисует результат:
- `fan.ts` — геометрия веера-дуги (наклон, теснота, попадание пальцем, раздвиг с прибитыми
  краями при драге).
- `flip.ts` — переворот карты/колоды как вращение на 180°/540° (не «схлопывание в ноль» —
  так карта не становится зеркальной), крен во время жеста, тянучка при запрещённом свайпе.
- `deckStack.ts` — раскладка стопки (передняя карта выше и правее, задняя ниже и левее —
  имитация света сверху справа), тень.
- `deckOrder.ts` — перестановки колоды: `moveCard`, `shuffleOrder`, `scatterCards`
  (выброшенные картой врезаются обратно на случайные места, остальные не трогаются),
  `isPermutationOf` (проверка, что клиент не подменил набор карт).
- `swipeShuffle.ts` — распознавание жеста-свайпа по скользящему окну скоростей (не по паре
  последних точек — иначе рывок в конце медленного драга читался бы как свайп).
- `handRow.ts` / `handView.ts` — раскладка приватной руки «шеренгой» и правила видимости
  чужих карт.
- `collapseButton.ts` — вписывание круглой кнопки «свернуть» в карман под дугой веера
  (радиус вычисляется, а не задаётся константой — иначе кнопка либо налезала на карты, либо
  висела в воздухе на других размерах экрана).
- `dealing.ts`, `dragMode.ts`, `dropZones.ts`, `selection.ts`, `barActions.ts` — очередь
  автораздачи, что можно тащить в каком режиме, зоны дропа, выделение элементов стола,
  какие две кнопки показывает нижняя панель.
- `deckFan.ts`, `topCard.ts`, `sortHand.ts`, `zoneLabels.ts`, `taunt.ts` — геометрия веера
  доски, какая карта стопки сверху, сортировка своей руки и то, как зона НАЗЫВАЕТСЯ в покое
  («стол», «сброс») против того, что она ОБЕЩАЕТ во время драга («сбросить», «взять себе»):
  подпись следует за тем, что у игрока в пальцах, а не только за самой зоной.

## Доска: стопки на столе

В раздаче (`phase: "lobby"`) стол не размечен вообще: колода лежит по центру, и
`centerZone` — это ВЕСЬ стол. «ГОУ!» размечает доску на боксы (`layout.ts`): `deckSlot`
слева, `centerZone` посередине, `discardSlot` справа. Отсутствующий бокс — это `null`,
а `dropZones.ts` превращает его в прямоугольник нулевого размера — поэтому и попадание,
и отрисовка отсекают его без единого частного случая.

- **`GameState.discard`** — сброс: карты, сыгранные со стола. Всегда лицом вверх (их уже
  сыграли, прятать нечего), последний элемент — верхняя карта. `discard_card` кладёт туда
  карту из руки, `take_discard` забирает обратно. `collect_hands` («Перераздача»)
  возвращает в колоду и сброс, и все руки.
- **Сброс лежит ГОРОЧКОЙ, а не стопкой** (`discardHeap.ts`): карты внахлёст под разными
  углами, свежесброшенная — в середину. Ровная стопка читается как колода, то есть как то,
  откуда БЕРУТ, — а из сброса в покое не берут, его сначала раскрывают веером. Узор
  ФИКСИРОВАННЫЙ, а не случайный: случайный пришлось бы хранить (иначе кучка
  перекладывалась бы на каждой перерисовке) и синхронизировать между игроками, чтобы у всех
  она выглядела одинаково. Рисуется семь карт и не больше — дальше силуэт не меняется,
  остальные считаются лежащими под ними (счётчик под слотом показывает настоящее число).
  В покое горка лежит РУБАШКОЙ ВВЕРХ: это «уже сыграно и убрано с глаз», а не витрина.
  Правило ПОКАЗА, а не состояние: на сервере карты сброса лежат лицом вверх, и прятать их
  по-настоящему незачем — любой может раскрыть веер и посмотреть.
- **Стопка доски раскрывается веером по тапу.** `BoardPile` (`engine/types.ts`) — какая
  стопка раскрыта у ЭТОГО зрителя; это локально, в отличие от `GameState.deckFanned`
  (веер дилера для слепой тасовки — он появляется и исчезает у всех разом). Любой веер
  доски раскрывается в `layout.boardFanAnchor` — центре игровой зоны — независимо от того,
  в каком слоте лежит сама стопка: так раскрытый веер не свисает с края экрана и всегда
  там, где его ждёт глаз.
- **У всех боковых элементов доски одна ширина** (`boardSlotWidth`): слот колоды, слот
  сброса и место бокового соседа читаются как одна колонка, приклеенная к краю экрана, и
  разнобой там читается как случайность, а не как разметка. Эталон — колода, единственная,
  чей размер продиктован содержимым. Сброс держит эту ширину и пустым: бокс размечает стол,
  а не показывает, сколько в нём карт.
- **Игральная зона** (`GameState.play`) — средний бокс в игре: СПИСОК КУЧЕК, в каждой от
  одной карты и сколько угодно, всё лицом вверх. Сервер хранит только состав, геометрии в
  нём нет: где кучка окажется на экране, решает `playGrid.ts` по её индексу. Правила — в
  `server/src/playRules.ts` (`play_card`, `take_play`, `clear_play`); зона ОБЩАЯ — класть и
  забирать может любой игрок из любой кучки, включая чужую карту. Очерёдность (queue lock)
  ляжет отдельным слоем сверху.
  - Опустевшая кучка исчезает из списка, и на сервере, и на клиенте. Пустая ячейка сетки
    место занимает, а показать ей нечего.
  - Устаревший индекс кучки не роняет действие — карта просто ложится новой кучкой. Из
    руки она визуально уже ушла, и отскок без объяснений хуже, чем «легла на ячейку левее».
  - В движке зона — ЧЕТВЁРТЫЙ `CardPile`, а не N штук: кучки разворачиваются в один
    плоский порядок (`playFlat.ts`), а раскладка переводит плоский номер обратно в «кучка
    k, j-я снизу». Именно это держит спрайты привязанными к идентичности карты, поэтому
    переезд карты между кучками играется перелётом, а не телепортацией.
  - Кучки зоны — обычные стопки доски под именем `play:N` (`engine/boardPile.ts`), и весь
    механизм веера достался им даром: тап раскрывает кучку в `boardFanAnchor`, ровно там
    же, где раскрываются колода и сброс. ЗАКРЫТАЯ кучка вдобавок отдаёт верхнюю карту
    обычным драгом: тянуться за веером на каждый ход — лишний шаг, а нужна чаще всего
    именно верхняя. Жесты не спорят: сдвинул палец — потащил карту, не сдвинул — раскрыл
    кучку (тап проверяет `dragHappened`).
  - Что делать с ОСТАЛЬНЫМИ картами стопки, когда с неё уводят карту, решается по стопке
    ЖЕСТА, а не по «раскрыт ли веер колоды». Там стояло `!this.deckFanned`, и драг из
    раскрытого сброса или кучки зоны (веер-то не колоды) молча пересобирал КОЛОДУ как n−1.
  - У кучки зоны СВОЯ геометрия (`playStack.ts`), а не колодная: колодный разъезд — про
    толщину пачки, а кучка — про то, что в ней лежит. Задние карты торчат из-под передней,
    вширь кучка не растёт больше чем в 1.2 карты (сетка тесная), а НИЖНЯЯ выпирает в
    правый нижний угол настолько, чтобы читался её угловой индекс — видно, что на дне, без
    раскрытия веера. Вертикальный разъезд больше горизонтального именно поэтому: угловой
    знак занимает нижнюю треть текстуры лица. Отсюда же ячейки сетки меряются ГАБАРИТОМ
    кучки, а не картой — иначе соседи налезали бы друг на друга ровно теми краями, ради
    которых разъезд и сделан. В отличие от колоды и сброса, кучка рисует ВСЕ свои карты:
    прятать их значило бы рисовать разъезд, которого не видно.
  - Пока карту тащат над зоной, стол ОТВЕЧАЕТ (`playHover.ts`): кучка под пальцем
    приподнимается и подрастает (её тень уходит дальше сама собой — подъём и есть
    превышение над покойным масштабом), а ближайшие соседи отступают, открывая её края.
    Подсветкой рамкой не обойтись: карта в драге сама накрывает ту кучку, к которой
    примеривается. Хит-тест при этом остаётся на НЕсдвинутой сетке — иначе цель дёргалась
    бы под неподвижным пальцем.
  - Сетка подбирает число колонок так, чтобы карта вышла крупнее всего. Кончается место —
    порядок уступок жёсткий: сжать карту до `PLAY_MIN_SCALE` и только потом прокрутка.
    Наоборот игрок листал бы стол уже на десятке кучек, хотя всё помещается чуть мельче.
    Место под СЛЕДУЮЩУЮ кучку резервируется всегда — иначе на забитой под край сетке новую
    кучку негде было бы начать.
- Посадка — «П» (`seatLayout.ts`): по бокам максимум по одному соседу (и всегда либо двое,
  либо ноль), остальные — в верхнюю прокручиваемую полосу. Боковые соседи стол по ширине НЕ
  режут — на телефоне это схлопнуло бы игровую зону в щель; вместо этого им уступают
  крайние боксы (колода съезжает ниже, сброс становится ниже ростом).

## Сеть: что правда, а что просто красиво

Жёсткое разделение, которое нельзя размывать при добавлении новых механик с колодой:

1. **Состояние — источник правды.** Порядок колоды и сторона каждой карты
   (`GameState.deck`, `GameState.faceUp`) ходят схемой Colyseus. Тяжёлые операции (тасовка,
   реордер) клиент считает **сам** и присылает уже готовый результат — сервер только
   проверяет, что это действительно перестановка того же набора карт (`isPermutationOf`),
   и не пересчитывает её заново. Это сделано ради мгновенного отклика: если бы клиент ждал
   эха с новым порядком, анимация тормозила бы на каждый пинг.
2. **`deck_fx` — только украшение.** Отдельная шина сообщений для эффектов (перевороты,
   растасовка), которые НЕ меняют состояние, а лишь показываются у зрителей. Сервер её не
   интерпретирует, только валидирует форму сообщения и ретранслирует.
3. **Ревизии против устаревшего эха.** `GameState.deckRev` — писать может только дилер;
   номер инкрементится на каждое действие. Клиент игнорирует входящее состояние, если его
   ревизия старше уже показанной локально — иначе собственное запоздавшее эхо откатывало бы
   картинку назад, и она бы «дёргалась». Тот же приём применён к своему порядку руки
   (`set_hand_order`): пока состав руки не изменился, клиент держит свой присланный порядок
   и не даёт чужому патчу состояния (кто-то нажал «Готов» рядом) перерисовать её заново
   несортированной.
4. **`ArraySchema.setAt` за пределы длины ДОПИСЫВАЕТ элемент**, а не пишет «в дырку»
   (массив длины 3 после `setAt(5, x)` становится длиной 4). Это конкретный источник бага
   «колода раздулась до 60 карт», который встречался дважды. Запись всей колоды всегда идёт
   как `clear()` + цикл `push()`, никогда `setAt` по всей длине.

## Правила видимости и роли

- **Открытая/закрытая рука** (`Player.handOpen`) — тумблер игрока: закрытая видна
  владельцу лицом, всем остальным рубашкой. Открытая видна всем так же, как владельцу.
- **Спрятанная карта** (`Player.handHidden`, императивный тумблер по конкретной карте) —
  не видна вообще никому, кроме владельца, даже при открытой руке.
- **Раздача включена всегда.** Колода лежит в центре стола рубашкой вверх: номинал ни одной
  карты не виден никому, включая дилера, пока она не попала кому-то в руку. Колоду трогает
  только дилер (тасовка, веер на столе, `deal_card`, автораздача, `reset_deck`,
  `collect_hands`); выйти из раздачи можно единственным способом — кнопкой «ГОУ!» (режим
  свободы ниже), а вернуться — «Перераздачей».
  Раньше был тумблер `dealMode`, и его выключение раскрывало всю колоду и превращало стол
  во вторую, докарточную механику: колода как один предмет, который таскают между центром,
  своей рукой и чужими местами, с переворотами. Тумблер и эта механика удалены — вместе с
  ними ушли `move_deck`, `flip_deck`, `flip_cards`, `reorder_deck`, а на клиенте — драг
  колоды целиком и геометрия «веер колоды в руке». Сама анимация переворота карты осталась:
  ею пользуются чужие эффекты и стороны карт с сервера, и на ней вырастут правила вроде
  «карта лежит на столе лицом».
- **Режим свободы** (`GameState.freeMode`, выключен, пока дилер не нажал «ГОУ!») — первый
  кирпич будущей системы правил (правила потом станут конфигами). Он переводит комнату в
  `phase: "playing"` БЕЗ раздачи колоды: колода остаётся в центре рубашкой вверх, и каждый
  тянет карты себе сам — `take_card` по умолчанию берёт верхнюю, но принимает ПОЗИЦИЮ,
  потому что колоду можно локально раскрыть веером и тянуть из него любую карту; `take_all`
  забирает всю колоду в руку. На нижней панели это кричалки «соснуть»/«сосать», а не кнопки
  со словом «Забрать»: только они влезают на экран 375px целиком (сам механизм укорачивания
  подписи по-прежнему меряет символы, а кнопка — пиксели, так что длинная подпись снова
  обрежется).
  Положить карту в чужую руку не может НИКТО, включая дилера: `deal_card` отвечает
  `action_rejected` с причиной `free_mode`. Двум одновременным «тяну» отдельная логика не
  нужна — Colyseus обрабатывает сообщения по одному, поэтому первому достаётся верхняя
  карта, второму следующая. Выход один — `collect_hands` («Перераздача»), она возвращает
  комнату в `lobby` и в раздачу. Побочный эффект смены фазы: обработчики колоды из
  `phase === "lobby"` (тасовка, `reset_deck`) отваливаются сами — так и задумано.
- **Вес голоса дилера** — 1.01 (`DEALER_VOTE_WEIGHT` в `handRules.ts`), не 1.5: дилер решает
  только равные голосования, двое обычных игроков его всегда перевешивают. Клиент обязан
  показывать ТОТ ЖЕ вес (`client/src/game/voteWeight.ts`) — раньше он рисовал 1.5, и полоска
  в баннере расходилась с реальным исходом.
- **Готовность как условие раздачи** — сервер не примет `deal_card` для игрока с
  `isReady === false` (кроме дилера — тот всегда готов). Так и задумано, хотя расходится с
  самой первой формулировкой задачи («пофиг готов или нет») — решение принято сознательно
  в процессе работы.
- Дилер имеет особые права: тасовка и раскрытие веера колоды на столе, ресет всей
  колоды и сбор карт обратно у всех игроков (`collect_hands`, `reset_deck`), автораздача по
  кругу от соседа (дилер получает последним).

## Известные компромиссы (сознательно, не забыто)

- `RoomEngine.ts` — всё ещё ~3400 строк, но это уже не стена: ~187 методов по 18 строк
  в среднем, самые длинные — `dropCard` (76) и `onPointerUp` (74). Большим его держат
  ~120 приватных полей, общих для жестов и анимаций; сжать дальше — значит вынести
  состояние из класса (отдельные владельцы жестов и анимаций), а это изменение крупнее
  всего сделанного и требует реальной причины.
- Обработчики серверных сообщений разложены по темам (`server/src/messages/*`) и берут
  у комнаты только то, что описано интерфейсом `RoomHost`; любая запись в схему идёт
  через `stateWrite.ts` — там же в одном месте соблюдается правило `clear()+push()`.
- Игровых правил (взятки, козыри, победные условия и т.п.) в коде нет — есть только
  механика владения и перемещения карт, на которую правила можно накладывать позже.
