## Находки

### 1. RoomEngine.ts (4713 строк) — единственный крупный файл, главная цель

236 методов, ~120 приватных полей, общих для жестов и анимаций. Разбить его безопасно
нельзя «в лоб» (вынести пачку методов в другой файл), потому что это ОДИН класс с глубокой
связностью полей. Наивный split уронил бы рантайм сессии — ровно то, чего просили избегать.

Кластеры методов (по префиксу): `on*` (28, обработчики указателя/колоды/руки/ховера),
`sync*` (15, синк визуала), `step*` (12, покадровые шаги анимаций), `start*`/`apply*` (по 9),
`build*` (7), `position*`/`poke*`/`make*`/`fly*`/`draw*` (4-5), `aim*`/`resolve*`/`drop*`.

**Предлагаемый поэтапный split (через КОМПОЗИЦИЮ, не переезд методов).** Каждый этап —
отдельный владелец состояния + класс, движок держит его как поле и делегирует. После
каждого этапа гонять оба сафети-нета (`RoomEngine.test.ts` на pixiFake) и `tsc`:

1. **`OverlayText`** — `rejectText`/`shout`/`taunt`/`notice` + их `build*`/`step*`/`show*`.
   Слабая связь с остальным (свои поля, свои тексты). Самый безопасный первый шаг.
2. **`ZoneView`** — `drawZones`/`zoneLabels`/`zoneHoverLabel`/`liveDropZones` +
   `resolveDrop`/`hoverZoneFor`. Уже почти чистое (zoneChrome/zonePaint снаружи).
3. **`FanInteraction`** — `poke*`/`glissando*`/`aim*`/`stepFanWiggle`/`applyFanWave` +
   поля `poke`/`hoverEnv`/`fanEnergy`/… Самый крупный и связный кластер, делать последним.
4. **`FlightAnimator`** — `fly*`/`playDealFlight`/`enqueueCardFlight`/`stepCardFlights` +
   `cardFlights`/`flightCards`/`seatFlightBias`.

Ожидаемый итог: RoomEngine ~ 4700 → ~2000-2500, каждый владелец 300-600.
**Риск: высокий, делать при живом контроле (утром), не unattended.**

### 2. Дубликаты-кандидаты (вынести в единый хендлер)

- **Тройка `flyCardOff`/`flyDiscardCardOff`/`flyPlayCardOff`** — идентичны, кроме шага
  «убрать карту из источника». Свести к `flyBoardCardOff(card, from, toId, source)` со
  свитчем по источнику. Маргинально по строкам, но убирает три имени. В горячем пути
  дропа — сделать при живом контроле.
- **Хвост дропа** — блоки `onDragChange(false); drawSeats(); drawZones(); positionDeckHit();
  wake()` повторяются в `dropCard` несколько раз. Свести в `finishDrag()`.
- Кандидат объединится с этапом 4 split'а (FlightAnimator).

### 3. Аудит цикличных эффектов (React) — риск багов сессии

Прошёлся по всем `useEffect` (App/AppMenu/Lobby/RoomScreen/room-хуки/use*-хуки).
**Бесконечных циклов не найдено.** Разбор рискованных:

- `RoomScreen` эффект сворачивания веера (`setMyBoardFan(null)` по условию) — СХОДИТСЯ:
  после сброса `myBoardFan===null` → `fannedCount=deck.length`, `fanNeedsDeck=false`;
  повторный `setMyBoardFan(null)` при том же значении React гасит (без ре-рендера).
- `useEngineEffect` (RoomCanvas) — односторонние сеттеры В движок, не в React-state,
  поэтому цикла в принципе нет.
- `useRoomState`/`useRoomSignals` — подписки на схему Colyseus, setState по приходу
  состояния (норма, драйвится сервером, не самозамыкается).

**Мелочь-ботлнек, не цикл:** в `RoomScreen` крутится `setInterval(100ms)` тик сессии
тасовки ВСЕГДА, даже когда никакой тасовки нет (tick просто возвращает `{send: null}`).
Можно гейтить по наличию активной сессии. Низкий приоритет.

### 4. Ботлнеки

- Рендер-цикл движка **спит в простое** (`canSleep`/`idleGate`) — по-крупному тут хорошо:
  ноль rAF/рендеров, когда ничего не движется. Явных горячих аллокаций на кадр в
  бодрствующем цикле не видно криминала (specs-массив теней собирается только при awake).
- Единственное «всегда крутится» — тот 100ms-таймер тасовки (см. выше).
- Отдельных тяжёлых мест не выявлено; при желании — профилировать на устройстве при
  большой колоде (52) на «полном» профиле анимаций.

### 5. Stale / неиспользуемое

- Клиент: по-настоящему мёртвых экспортов НЕТ (ts-prune чист по «не-в-модуле»).
- Сервер: `RejectReason` был мёртв — **исправлено** (оживлён в отправке).
- Тест-онли экспорты (типы `*Input`/`*Options`/`*Result`) — это интерфейсы аргументов
  чистых функций, они «used in module» и нормальны; удалять не нужно.
