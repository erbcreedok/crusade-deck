# Crusade Deck — контекст проекта

## Что это
Мультиплеерная карточная игра для мобильных браузеров. 2–32 игрока в сессии,
выбор колоды 36 или 52 карты. Пиксельный кэжуал-стиль (Balatro-style), не
литеральный "стол с людьми" — виртуальный круглый стол, места по эллипсу.

## Стек
- **Сервер**: Colyseus (Node.js), свои аккаунты (recovery-код) вместо Firebase
- **Клиент**: React + Vite + Framer Motion
- **Auth**: свои аккаунты (короткий recovery-код, без пароля) — основной путь.
  Firebase Authentication (Google, Apple, email/password, magic link) —
  задел на будущее в коде (`client/src/firebase.ts`, `server/src/auth.ts`),
  реально не настроен и не используется, пока ключи не подставлены.
- **Тесты**: vitest в обоих пакетах (`npm test` в `server/` и `client/`)
- **Хостинг**: self-hosted Colyseus на VPS в Казахстане (hoster.kz, ДЦ Алматы)
  за Caddy — конфиги и шпаргалка в `deploy/`
- **Возможно позже**: Redis для горизонтального масштабирования

## Что уже готово

### Сервер (Colyseus)
- `CardRoom`: лобби (ready), дилер + голосования (propose_dealer/propose_kick/vote,
  вес дилера 1.5, таймаут голосования 10с — не проголосовавшие не считаются),
  паблик/приват (может переключить любой игрок), реальная колода
  (36/52, Fisher-Yates тасовка, авто-раздача поровну всем подключённым)
- Схемы `GameState` / `Player` / `Proposal` (`@colyseus/schema`)
- Свои аккаунты (`server/src/accounts.ts`) — recovery-код, без Firebase
- Верификация Firebase-токенов на сервере готова, но не задействована в
  реальном флоу (клиент всегда шлёт `accountId`, не Firebase-токен)
- Реестр 6-значных инвайт-кодов для приватных комнат + список паблик-комнат
- Тесты: `*.test.ts` рядом с исходниками, интеграционные для `CardRoom` через
  `@colyseus/testing` (реальный локальный сервер + клиент)
- Проходит `tsc --noEmit` без ошибок

### Клиент (React/Vite)
- Вход: свой аккаунт (recovery-код) — "новый профиль" или "восстановить по коду"
- Единое меню (☰): профиль (имя, recovery-код + обновление), тумблер анимаций
  (гасит и Framer Motion, и CSS-фон), паблик/приват, выход из комнаты
- Лобби: список публичных комнат + вход по коду
- Стол: круглая рассадка, своя рука веером лицом вверх, чужие — рубашкой вниз
  со счётчиком карт, баннер голосования с обратным отсчётом
- iOS/мобильный фуллскрин без скролла и зума (viewport + CSS фиксы)
- Тесты: vitest + Testing Library для чистой логики (хуки, парсинг карт)
- Проходит `tsc --noEmit` без ошибок

## Тестирование и TDD
С 2026-07-22 TDD — основной способ разработки в этом проекте: **новый код
(фича или багфикс) начинается с падающего теста**, потом реализация до
зелёного. Не откладывать тесты "на потом".

- Сервер: `cd server && npm test` (vitest). Чистую логику (`accounts.ts`,
  `inviteCodes.ts`, `publicRooms.ts`) — юнит-тестами. `CardRoom.ts` —
  интеграционными тестами через `@colyseus/testing` (`boot()` +
  `createRoom()`/`connectTo()`), с реальным локальным сервером/клиентом.
  `VOTE_TIMEOUT_MS` читается через `process.env` в момент вызова (не при
  загрузке модуля) специально для тестируемости коротких таймаутов.
- Клиент: `cd client && npm test` (vitest + jsdom + Testing Library).
  Приоритет — детерминированная логика (хуки, парсинг), а не визуальные/
  анимационные детали компонентов.

## Чего нет / что дальше
1. Игровые правила и слой `GameEngine` — ещё не написаны вообще (есть только
   раздача карт, самих правил игры нет)
2. Раздача карт — сделан только первый срез (реальная колода, тасовка,
   авто-раздача поровну). Отдельными итерациями ещё предстоят: тасовка
   кликами/щелчками по колоде, драг колоды игроку для среза, ручная раздача
   по одной карте, тоггл веер/стопка и скрытие своей руки, звуки
3. Деплой — конфиги готовы (`deploy/`), сам сервер ещё не заказан
4. Опционально — Redis для скейлинга

## Важные решения и почему
- Инфраструктура сознательно строится раньше игровой логики — это осознанный
  выбор последовательности, не забыли про правила игры, это следующий шаг.
- Латентность ~100–150ms приемлема для карточной пошаговой игры → удалённые,
  но производительные серверы (например Hetzner) ok, даже если физически
  далеко от Казахстана — если только не важна локальная юрисдикция.
- Framer Motion выбран специально под Balatro-style взаимодействие с картами
  (наклон при hover, физика drag со spring, веерная раскладка руки в hand) —
  не дефолт, а осознанный UX-выбор.
- Свои аккаунты вместо Firebase — осознанный выбор для простоты локальной
  разработки (см. память проекта); Firebase остаётся задачей на будущее,
  когда понадобится реальный Google/Apple вход.
- Пользователь (я) — в Казахстане, поэтому вместо Hetzner выбран местный
  хостинг: hoster.kz Cloud 1-2-50 (4 800 ₸/мес, ДЦ Алматы) + домен .com
  там же. Латентность ~5–15 мс вместо ~80–120 мс до Европы, оплата в тенге.
  Рассмотрены и отклонены: PS.kz (конфиги тарифов скрыты за формой заказа),
  Serverspace.kz и oblako.kz (вдвое дороже при тех же характеристиках).
- Клиент и сервер за одним Caddy на одной машине — один origin, никакого
  CORS, TLS от Let's Encrypt автоматом. Домен нужен не только ради `wss://`,
  но и ради secure context: без него отваливается `navigator.clipboard`,
  то есть кнопка «скопировать recovery-код».
- Адреса сервера не зашиты в сборку клиента, а выводятся из `window.location`
  (`client/src/serverUrl.ts`) — смена домена не требует пересборки.
  Подвох: сокет Colyseus живёт в корне (`/{processId}/{roomId}`), рядом со
  статикой, поэтому в Caddy он ловится матчером по заголовкам апгрейда,
  а не по префиксу пути.
